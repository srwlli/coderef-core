# index.json — Authoritative Documentation

## Executive Summary

`index.json` is a generated JSON array file containing a flat index of all code elements discovered by the CodeRef scanner. It stores 1,270 code elements (functions, classes, methods, etc.) with metadata including type, name, file path, line number, and export status. The file serves as the primary element registry for codebase analysis, query operations, and context generation. It is automatically generated by the scanner module and should not be manually edited.

## 1. Architecture Overview

The index.json file is the output artifact of the CodeRef scanning process:

```
Scanner Module → scanCurrentElements() → ElementData[] → index.json
```

**File Structure:**
- **Format:** JSON Array
- **Location:** `.coderef/index.json` (relative to project root)
- **Size:** ~8,892 lines, 1,270 elements
- **Encoding:** UTF-8
- **Indentation:** 2 spaces

**Integration Points:**
- **Generated by:** `scanner.ts` → `scanCurrentElements()` function
- **Consumed by:** Query executor, context generator, export tools
- **Schema:** `ElementData` interface from `types.ts`

## 2. State Ownership & Source of Truth

| State | Owner | Type | Persistence | Source of Truth |
|-------|-------|------|-------------|-----------------|
| Element index | Scanner Module | System | File system | `index.json` file |
| Element data | Scanner Module | Domain | File system | Array items in JSON |
| File metadata | Scanner Module | System | File system | File modification time |
| Element count | Scanner Module | System | File system | Array length |

**Precedence Rules:**
- Scanner output is authoritative (file is regenerated on each scan)
- Manual edits are overwritten (file is write-only from scanner perspective)
- File modification time indicates last scan timestamp
- Element order is non-deterministic (may vary between scans)

## 3. Data Persistence

**Storage:**
- **Location:** `.coderef/index.json` (project root relative)
- **Format:** JSON array, UTF-8 encoding
- **Backup:** Not automatically backed up (consider git exclusion)

**Versioning:**
- **No version field:** File does not contain version metadata
- **Implicit versioning:** File modification time indicates generation time
- **Regeneration:** File is completely rewritten on each scan (no incremental updates)

**Failure Modes:**
- **Write failure:** Scanner throws error, file may be incomplete or missing
- **Read failure:** Consumers should handle missing file gracefully
- **Corruption:** File may be invalid JSON if write interrupted

**Recovery:**
- **Regeneration:** Re-run scanner to regenerate file
- **Validation:** Consumers should validate JSON structure before parsing
- **Fallback:** Empty array `[]` if file missing or invalid

## 4. State Lifecycle

1. **Initialization:** Scanner creates `.coderef/` directory if missing
2. **Scanning:** Scanner analyzes source files and collects elements
3. **Serialization:** Element array serialized to JSON with 2-space indentation
4. **Writing:** JSON written to `index.json` file (overwrites existing)
5. **Validation:** File can be validated by parsing as JSON
6. **Consumption:** Consumers read and parse file for element queries
7. **Regeneration:** File overwritten on next scan operation

## 6. Event & Callback Contracts

| Event | Trigger | Payload | Side Effects |
|-------|---------|---------|--------------|
| File generation | Scanner scan | `elements: ElementData[]` | Creates/overwrites `index.json` |
| File read | Consumer access | None | Parses JSON, returns array |
| File validation | Consumer validation | None | Validates JSON structure |

**ElementData Schema Contract:**
```typescript
interface ElementData {
  type: string;           // 'function' | 'class' | 'method' | 'interface' | etc.
  name: string;           // Element name (e.g., 'convertGraphToElements')
  file: string;           // Absolute file path
  line: number;          // Line number (1-indexed)
  exported?: boolean;     // Whether element is exported (default: false)
  parameters?: string[]; // Function parameters (if applicable)
  calls?: string[];      // Called functions (if applicable)
  metadata?: Record<string, any>; // Additional metadata
}
```

## 7. Performance Considerations

**Known Limits:**
- **File size:** ~8,892 lines, ~1,270 elements (current instance)
- **Parse time:** ~50-100ms for JSON parsing (1,270 elements)
- **Memory:** ~500KB-1MB in memory when parsed

**Bottlenecks:**
- **Large codebases:** Files with >10,000 elements may take 200-500ms to parse
- **JSON parsing:** Native `JSON.parse()` is efficient but synchronous
- **File I/O:** Reading large files may block event loop

**Optimization Opportunities:**
- **Streaming parser:** For very large files (>50MB)
- **Indexed access:** Build secondary indexes for faster lookups
- **Compression:** Gzip compression for storage (not implemented)

**Deferred Optimizations:**
- **Incremental updates:** Currently full rewrite (not incremental)
- **Binary format:** JSON is human-readable but not optimal for large datasets
- **Caching:** Parsed results not cached (re-parsed on each access)

## 11. Common Pitfalls & Sharp Edges

**File Overwrite:**
- File is completely overwritten on each scan (no merge)
- Manual edits are lost
- Always regenerate from source, don't edit manually

**Path Format:**
- File paths are absolute (e.g., `C:/Users/...`)
- Paths may vary between systems (Windows vs Unix)
- Consumers should normalize paths for cross-platform compatibility

**Element Order:**
- Element order is non-deterministic
- Don't rely on array index for element identity
- Use `name` + `file` + `line` as unique identifier

**JSON Structure:**
- File must be valid JSON (no comments, no trailing commas)
- Invalid JSON causes parse errors
- Always validate before parsing

**Missing Properties:**
- `exported` is optional (defaults to `false` if missing)
- `parameters` and `calls` are optional (only for applicable elements)
- Consumers should handle missing properties gracefully

**File Size:**
- Large codebases may generate very large files (>10MB)
- JSON parsing may be slow for very large files
- Consider streaming or indexed access for large files

**Regeneration:**
- File is regenerated on each scan
- Don't cache file contents across scans
- Check file modification time to detect updates
