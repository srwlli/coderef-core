# graph.json — Authoritative Documentation

## Executive Summary

`graph.json` is a generated JSON object file containing a dependency graph representation of codebase relationships. It stores graph nodes (code elements), edges (relationships), and metadata (scan information). The file serves as the source of truth for dependency analysis, relationship queries, and impact simulation. It is automatically generated by the analyzer module and should not be manually edited. The current instance contains 1,270 nodes and 0 edges (empty edges array indicates no relationships detected or graph not fully populated).

## 3. Data Persistence

**Storage:**
- **Location:** `.coderef/graph.json` (project root relative)
- **Format:** JSON object, UTF-8 encoding
- **Backup:** Not automatically backed up (consider git exclusion)

**Versioning:**
- **No version field:** File does not contain version metadata in graph structure
- **Implicit versioning:** File modification time indicates generation time
- **Metadata timestamp:** `metadata.scanned_at` provides ISO8601 timestamp
- **Regeneration:** File is completely rewritten on each analysis (no incremental updates)

**Failure Modes:**
- **Write failure:** Analyzer throws error, file may be incomplete or missing
- **Read failure:** Consumers should handle missing file gracefully
- **Corruption:** File may be invalid JSON if write interrupted
- **Empty edges:** Edges array may be empty if relationships not detected

**Recovery:**
- **Regeneration:** Re-run analyzer to regenerate file
- **Validation:** Consumers should validate JSON structure before parsing
- **Fallback:** Empty graph `{nodes: [], edges: [], metadata: {}}` if file missing or invalid

## 4. State Lifecycle

3. **Graph Construction:** Analyzer builds dependency graph from imports and calls
4. **Serialization:** Graph object serialized to JSON with 2-space indentation
5. **Writing:** JSON written to `graph.json` file (overwrites existing)
6. **Validation:** File can be validated by parsing as JSON

**Graph Schema Contract:**
```typescript
interface DependencyGraph {
  nodes: GraphNode[];        // Array of graph nodes
  edges: GraphEdge[];        // Array of graph edges (relationships)
  metadata: {                // Scan metadata
    languages: string[];      // Scanned languages (e.g., ['ts', 'tsx', 'js', 'jsx'])
    analyzer: string;         // Analyzer type (e.g., 'ast')
    element_count: number;   // Total element count
    scanned_at: string;      // ISO8601 timestamp
  };
}

interface GraphNode {
  id: string;                // Unique node ID (format: "file:line:name")
  type: string;              // Element type (e.g., 'function', 'class', 'method')
  name: string;              // Element name
  file: string;              // Absolute file path
  line: number;              // Line number (1-indexed)
  metadata?: Record<string, any>; // Additional node metadata
}

interface GraphEdge {
  source: string;             // Source node ID
  target: string;             // Target node ID
  type: string;               // Edge type (e.g., 'imports', 'calls', 'depends-on')
  weight?: number;            // Edge weight (optional)
  metadata?: Record<string, any>; // Additional edge metadata
}
```

## 7. Performance Considerations

**Known Limits:**
- **File size:** ~8,907 lines (current instance)
- **Node count:** 1,270 nodes (current instance)
- **Edge count:** 0 edges (current instance - may vary)
- **Parse time:** ~50-150ms for JSON parsing (1,270 nodes)
- **Memory:** ~1-2MB in memory when parsed

**Bottlenecks:**
- **Large graphs:** Graphs with >10,000 nodes may take 500ms-1s to parse
- **Dense graphs:** Graphs with >100,000 edges may be slow to traverse
- **JSON parsing:** Native `JSON.parse()` is efficient but synchronous
- **File I/O:** Reading large files may block event loop

**Optimization Opportunities:**
- **Streaming parser:** For very large files (>50MB)
- **Indexed access:** Build node/edge indexes for faster lookups
- **Compression:** Gzip compression for storage (not implemented)
- **Graph algorithms:** Optimize traversal for large graphs

**Deferred Optimizations:**
- **Incremental updates:** Currently full rewrite (not incremental)
- **Binary format:** JSON is human-readable but not optimal for large graphs
- **Caching:** Parsed results not cached (re-parsed on each access)
- **Lazy loading:** Load nodes/edges on-demand (not implemented)

## 11. Common Pitfalls & Sharp Edges

**File Overwrite:**
- File is completely overwritten on each analysis (no merge)
- Manual edits are lost
- Always regenerate from source, don't edit manually

**Empty Edges:**
- Edges array may be empty if relationships not detected
- Current instance has 0 edges (may indicate incomplete analysis)
- Consumers should handle empty edges gracefully
- Check `metadata.element_count` to verify analysis completion

**Node ID Format:**
- Node IDs use format: `"file:line:name"` (e.g., `"C:/path/to/file.ts:42:functionName"`)
- IDs are unique identifiers for nodes
- Don't parse IDs manually, use provided node properties

**Path Format:**
- File paths are absolute (e.g., `C:/Users/...`)
- Paths may vary between systems (Windows vs Unix)
- Consumers should normalize paths for cross-platform compatibility

**JSON Structure:**
- File must be valid JSON (no comments, no trailing commas)
- Invalid JSON causes parse errors
- Always validate before parsing

**Metadata Timestamp:**
- `metadata.scanned_at` is ISO8601 format (e.g., `"2025-12-23T02:48:16.425659"`)
- Timestamp indicates when graph was generated
- Use for cache invalidation and freshness checks

**Graph Completeness:**
- Graph may be incomplete if analysis interrupted
- Check `metadata.element_count` matches expected count
- Empty edges may indicate analysis not fully completed

**Regeneration:**
- File is regenerated on each analysis
- Don't cache file contents across analyses
- Check file modification time or `metadata.scanned_at` to detect updates

**Edge Types:**
- Edge types: `'imports'`, `'calls'`, `'depends-on'`, `'implements'`, `'tests'`, `'reexports'`
- Edge type determines relationship semantics
- Consumers should handle unknown edge types gracefully
